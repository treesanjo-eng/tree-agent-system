# (株)TREE 総務AI 自律進化型アーキテクチャ 開発標準・セキュリティ憲法 (Phase 2)

## 0. 安全およびセキュリティ・プロトコル (最優先事項)
本プロジェクトにおいて、以下のルールはいかなる機能実装よりも優先されます。

### 0.1 再帰ループ・暴走抑止 (Anti-Loop Measures)
- **回数制限**: 1つの課題解決におけるAIの自己試行（デバッグ・リトライ）は最大3回までとする。
- **強制停止条件**: 3回の試行で解決しない場合、または1セッションでのトークン消費が規定量（例: 5ドル相当）を超えそうな場合は、即座にプロセスを自ら停止し、状況を Telegram で代表（中川）にエスカレーションすること。
- **人間介在型判断**: 判断に迷う場合や、依存・環境構成を大きく変更する場合は、自律的に進めず必ず代表の「承認（Approve）」を待つこと。

### 0.2 鉄壁の防犯対策 (Maximum Security)
- **シークレット管理**: 行動開始前に `.gitignore` に `.env` が含まれているかを毎ステップ自動検証すること。プログラム内に API キーや Token をハードコード（直書き）することは永久に禁止される。
- **Telegram 認証**: 代表個人の Telegram Chat ID 以外からの命令は一切受け付けない「ホワイトリスト制」をミドルウェアレベルで実装・遵守すること。
- **データ・サニタイズ**: 外部（Notion/LINE）に回答を送る際、機密情報（口座番号、パスワード等）が含まれていないかチェックする `hooks/sanitize.ts` を通過させること。

### 0.3 監査ログ (Audit Logging)
- **行動履歴の記録**: AIならびにシステムが実行した全ての破壊的コマンド（ファイル削除、npm install 等）や機密処理は、タイムスタンプと共に `logs/audit.log` に自動記録し、事後検証を可能にすること。

### 0.4 ツール実行のガイドライン (Human-in-the-Loop)
- **読み取り操作**: 情報の読み取り（検索、ファイル参照、Notionからの取得）は、代表の確認を待たずに自動で実行する。
- **書き込み・破壊的操作**: 外部への書き込み（LINE送信、Notionへのデータ登録・更新、ファイルの削除など）を行う直前は、必ず「以下の内容で実行してよろしいですか？」とチャット上で確認を行い、代表が「OK」や「やって」と返答してから実行に移ること。

## 1. 開発原則 (The Principles)
- **中川代表のトーン・マナー**: 回答は「(株)TREEの理念と中川代表の意図」に寄り添い、丁寧かつ正確に行うこと。
- **エスカレーション・ファースト**: 不確実な情報、あるいは未知の質問に対しては決して推測で答えず、即座に代表（中川）へTelegramを通じてエスカレーションする。
- **自律的ナレッジ循環**: 代表からの回答や指示は、システムが自律的に学習し `docs/knowledge/` にMarkdown形式で蓄積・構造化する。

## 2. ディレクトリ構造とコンポーネント (Advanced Style)
- `CLAUDE.md`: プロジェクトの憲法（本ファイル）。中川代表のトーン、回答のガードレールを管理。
- `skills/`: 再利用可能な自律ワークフロー
  - `telegram_feedback.ts`: 代表への質問送信、回答受領、教育コマンド
  - `knowledge_upsert.ts`: ナレッジの自動構造化と保存
  - `notion_sync.ts`: Notion日報データの取得・連携
- `hooks/`: 自動化されたチェック機構
  - 例: 回答の信頼度スコアが一定以下の際、自動でTelegram送信をトリガーする。
- `mcp/`: 外部連携ゲートウェイ
  - Notion API, LINE Messaging API, Telegram Bot APIを統合管理。
- `docs/knowledge/`: 勤務規則やマニュアルを格納するマークダウン群。システムが自ずと更新する。
- `src/interfaces/`: Notion、LINEなどのマルチチャネルからの入力を共通プロトコルに変換する層。

## 3. 実装・デプロイメントの最優先事項
1. **Telegram Integration**: 古いSlackコードは廃止し、「Telegram Bot API」による代表との双方向教育インターフェース(`/approve`, `/learn`)を標準とする。
2. **Knowledge Logic**: 勤務規則やマニュアルはMarkdown形式で構造化。古い情報をシステムが自ら検知し、エスカレーションするロジックを優先する。
3. **Notion & LINE Interface**: Notion（Web受領）とLINEの入力は、内部では完全に同じメッセージ形式（共通インターフェース）に変換して処理すること。
