# TREE日報アシスタント アーキテクチャ設計根拠

## 1. 概要
本プロジェクトは、単なるSlackボットではなく、将来的に「自律的にエージェントを生成・展開するメタエージェント」の標準テンプレートとなるべく設計されている。

## 2. 設計原則：疎結合とモジュール化
高い拡張性と自動生成の容易さを確保するため、以下の3つの主要なレイヤーに分離する。

### A. Config レイヤー (`src/config/`)
- **役割**: 環境変数や静的設定の管理。
- **根拠**: ロジックから設定を完全に切り離すことで、異なる組織や用途向けにクローン（生成）する際、設定ファイルのみを置き換えれば動作するようにする。

### B. Core レイヤー (`src/core/`)
- **役割**: ビジネスロジック、日報収集アルゴリズム、データ処理。
- **根拠**: インターフェース（Slackなど）に依存しない純粋なロジックを保つ。将来、DiscordやWeb UIへ拡張、あるいはLLMエンジンを変更する際も、コア部分を流用可能にする。

### C. Interfaces レイヤー (`src/interfaces/`)
- **役割**: 入出力のゲートウェイ（Slack Boltなど）。
- **根拠**: 外部プラットフォームとの接点をカプセル化する。自動生成エージェントが「どの口（Slackかメールか）」を持つかを選択可能にする。

## 3. 次世代エージェントへの布石
本テンプレートに従って開発されたコードは、LLMがパースしやすいファイル構造とコメント規約を持つ。これにより、「新しいエージェントを作る」という命令に対し、メタエージェントが本ディレクトリ構造を雛形としてコピーし、必要なロジックを注入するプロセスを自動化できる。
